# Rei 方案设计

> 基于 opencode，对**任意路径下的本地代码仓库 + 多级 markdown wiki** 提供三条能力：
> 1. 代码 → Wiki
> 2. 需求端到端实现（结合 wiki + 代码库 → 产出代码 + 文档）
> 3. 根据 wiki + 代码库对新代码或仓内代码做 BUG 定位
>
> 成品接线（不自造）：**code-review-graph**（CRG，代码图 + 影响面 + wiki 生成）+ **opencode-dcp**（DCP，上下文兜底压缩）。CRG 一身二任：当 code_graph_review，又当 DeepWiki（`wiki` 子命令出 markdown）。
>
> 状态：**已落地**。`.opencode/`（8 agent + 6 command）+ `opencode.json` + `AGENTS.md` 已写；CRG / DCP 已装。

---

## 0. 设计基线

- **成品优先、不自造**：DeepWiki / code_graph_review / DCP 一律用现成成品，不手写方法学 skill。
- **配置优先**：能用 opencode 原生 agents / commands / references / permissions 解决就不写插件（DCP 是唯一插件，因其本身就是 opencode 插件）。
- **本地/隐私**：CRG 本地运行（SQLite 图库 + Leiden 社区，零遥测；embeddings / LLM 均 opt-in）；DCP 本地插件；代码不外发。
- **人在 loop**：设计、提交两处硬检查点。
- **权限最小化**：每个 subagent 只给其职责所需权限。
- **经验沉淀进 wiki**，不灌 `AGENTS.md` / skills。

---

## 1. 核心思想：Wiki 导航层 + 代码真相层

把 **wiki** 与 **代码库**当一个**双层索引**协同用——「结合 wiki 与代码库」的主线：

| 层 | 角色 | 特性 | 加载 |
|---|---|---|---|
| Wiki（多级 md） | 导航/语义层：what/why/架构/约定 | 小、稳、便宜 | 先读索引页，按需深读 |
| 代码库 | 真相层：how/实现 | 大、易变、贵 | 经 wiki 的 `source_paths` 锚点 + CRG 代码图，**图引导选择性加载** |

双向锚点：wiki frontmatter `source_paths` → 代码位置；wiki 页之间双链。**三条流都先读 wiki 拿地图，再下钻代码，下钻由 CRG 的图引导只加载相关文件。**

### 两层上下文策略

- **日常层（CRG）**：代码图引导「只加载相关文件」+ CRG 的 `detect-changes --brief` / `visualize --format json` 给紧凑影响面与图，从源头压低上下文体量。
- **兜底层（DCP 全局插件）**：上下文逼近上限时自动压缩（替换为摘要占位，不改会话历史），多数时候 dormant。

> DCP **不是**用来选代码的；选代码由 CRG 图引导。DCP 只在上下文快炸时介入。

## 2. 三条工作流

### WF1 — 代码 → Wiki
- **入**：`<repo-path>`、`<wiki-path>`（默认 `<repo>/docs/wiki`）
- **流**：`dev` → `cartographer`（`code-review-graph build --repo <R>` 建图）→ `wiki-writer`（`code-review-graph wiki --repo <R>` 生成，再同步到 `<wiki-path>` + 补索引）→ 覆盖前 diff 既有 wiki 供审
- **出**：多级 markdown wiki（落在 `<wiki-path>`）

### WF2 — 需求端到端
- **入**：需求文本；repo = cwd 或 `--repo`；wiki 路径
- **流**：`dev` → `planner`（读 wiki + CRG 图定位触点 → 设计：改动点/风险/测试计划）→ 🛑检查点1 → `coder`（按 repo `AGENTS.md` 实现）→ `reviewer`（checklist + `detect-changes --brief` 影响面）→ 循环 → `tester`（gates，按仓发现）→ 循环 → `wiki-writer`（更新受影响 wiki 页）→ `stage` → 🛑检查点2
- **出**：代码改动 + 更新后的 wiki + 变更说明

### WF3 — BUG 定位
- **入**：bug 报告 / 失败代码 / 新代码；repo；wiki
- **流**：`dev` → `bug-tracer`（读 wiki 拿预期行为 + 架构 → CRG 图沿数据/控制流从症状回溯根因 → 根因假设 + 定位代码 + 证据 + 修复建议）→ 🛑报告供审 →（可选）`coder` 施修 → `tester` → `stage`
- **出**：BUG 报告（根因/证据/修复建议）+ 可选修复改动

> 三流都是长任务，可能撑爆上下文——**这才是 DCP 兜底的场景**。

## 3. 成品接线（跨流复用）

| 成品 | 角色 | 形态 | 装法 |
|---|---|---|---|
| **code-review-graph**（CRG） | code_graph_review（图/影响面）+ DeepWiki（`wiki` 出 markdown） | CLI `code-review-graph`（`~/.local/bin/`），`--repo <R>` 任意路径 | `uv tool install code-review-graph`（已装 v2.3.7） |
| **opencode-dcp**（DCP） | 上下文兜底压缩 | 全局 opencode 插件，自动 | `opencode plugin @tarquinen/opencode-dcp@latest --global`（已装） |

CRG 子命令（agent 用）：`build` / `update` / `status` / `detect-changes [--brief]` / `wiki` / `visualize --format json` / `serve`（MCP，可选）。本地 SQLite 图库 `.code-review-graph/`，零遥测；embeddings / LLM 均 opt-in（指向 localhost 免告警）。

> 不再自造 deep-wiki / code-graph-review / dcp skill——全由成品覆盖。

## 4. Agent 拓扑 + 权限

primary `dev` 编排 + 人在 loop；其余 `mode: subagent`，最小权限（bash 对 `git *` / `code-review-graph *` 放行，禁 `git commit/push`、`rm`）。

| Agent | mode | 职责 | 关键权限 |
|---|---|---|---|
| `dev` | primary（默认） | 编排三流、检查点、变更说明 | edit: allow, bash: allow（禁 commit/push） |
| `cartographer` | subagent | CRG 建图 + 导出摘要 | edit: deny, bash: 只读 git + CRG |
| `wiki-writer` | subagent | CRG `wiki` 出页 + 同步到 wiki 根 | edit: allow（**仅 wiki 根下**）, bash: 只读 git + CRG |
| `planner` | subagent | 读 wiki+图，出设计 | edit: deny, bash: 只读 git + CRG |
| `coder` | subagent | 实现 | edit: allow, bash: allow（禁 commit/push） |
| `reviewer` | subagent | 自审 + CRG 影响面 | edit: deny, bash: 只读 git + CRG |
| `tester` | subagent | 跑 gates | edit: deny, bash: allow |
| `bug-tracer` | subagent | CRG 图回溯根因、定位 | edit: deny, bash: 只读 git + CRG |

落地：`.opencode/agents/<name>.md`，frontmatter 带 `mode/permission/description`。

## 5. Wiki 结构与 frontmatter

CRG `wiki` 按 Leiden「代码社区」生成 markdown 页（落目标仓 `.code-review-graph/wiki/`）；`wiki-writer` 再同步到用户指定的 `<wiki-path>`，并补 `README.md` 索引 + `index.md`。frontmatter 目标字段：

```yaml
id / title / level / parent / related(双链) / source_paths(→代码) / last_sync_commit
```

> wiki 根建议结构：`README.md`（入口）+ `architecture/`（L1）+ `modules/<mod>/`（L2/L3）+ `decisions/`（ADR）+ `index.md`。CRG 原生页以社区划分；`wiki-writer` 负责对齐到上述约定。

## 6. Commands（`.opencode/commands/*.md`）

| 命令 | 作用 | agent |
|---|---|---|
| `/wiki <repo> [wiki]` | WF1 结构 wiki（CRG 原始页） | dev |
| `/wiki-prose <repo> [out] [community]` | 架构 wiki（DeepWiki 风格，知识库级，wiki-writer 读源码写） | dev |
| `/feature <需求> [--repo] [--wiki]` | WF2 | dev |
| `/bug <报告\|路径> [--repo] [--wiki]` | WF3 | dev |
| `/review` | 当前改动自审 | dev |
| `/gates` | build/lint/typecheck/test（按仓发现） | dev |
| `/stage` | 本地改动+变更说明 | dev |

## 7. 配置（`opencode.json`）

- `$schema`、`default_agent: "dev"`、`instructions: ["AGENTS.md"]`
- `permission`：`edit: ask`；`bash: { *: ask, git *: allow, code-review-graph *: allow, git commit/push: deny, rm *: deny }`；`external_directory: allow`
- `compaction: { auto: true, tail_turns: 15 }`（opencode 自带压缩；DCP 插件在其上做更智能的摘要兜底）
- `model` / `small_model` 由全局配置提供（本仓不写死）
- DCP 全局插件已写进 `~/.config/opencode/opencode.jsonc`（`plugin: ["@tarquinen/opencode-dcp@latest"]`）

## 8. 「任意路径」机制

- 工具集在源仓 `/home/dlrow_hl/Rei` 版本化；命令以 `<repo>` / `[wiki]` 路径参接任意目标仓。
- CRG 所有子命令带 `--repo <target>`，不依赖 cwd；agent 用绝对路径操作目标仓，`external_directory: allow` 放行。
- 重复操作某仓 → 在该仓丢一个 mini `opencode.json`（`references.wiki` + `instructions` 指向该仓 `AGENTS.md`），工具集自动套用。
- （可选）要解锁 CRG 的 30 个 MCP 工具（`query_graph`/`traverse`/`semantic_search`…），可在目标仓跑 `code-review-graph install` 让它写 OpenCode MCP 配置；CLI 模式已覆盖三流主路径。
