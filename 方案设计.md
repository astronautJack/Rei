# Rei 方案设计

> 基于 opencode，对**任意路径下的本地代码仓库 + 多级 markdown wiki** 提供三条能力：
> 1. 代码 → Wiki
> 2. 需求端到端实现（结合 wiki + 代码库 → 产出代码 + 文档）
> 3. 根据 wiki + 代码库对新代码或仓内代码做 BUG 定位
>
> 成品接线（不自造）：**code-review-graph**（CRG，代码图 + 影响面 + 结构 wiki）+ **opencode-dcp**（DCP，上下文兜底压缩）。CRG 出**结构页**；**散文架构文档**由 `wiki-writer` agent 读结构页+下钻源码自己写。
>
> 状态：**已落地**。`.opencode/`（8 agent + 7 command）+ `opencode.json` + `AGENTS.md` 已写；CRG / DCP 已装并按 glm-5.2 窗口调好阈值。

## 0. 设计基线

- **成品优先、不自造**：DeepWiki / code_graph_review / DCP 一律用现成成品，不手写方法学 skill。
- **配置优先**：能用 opencode 原生 agents / commands / references / permissions 解决就不写插件（DCP 是唯一插件，因其本身就是 opencode 插件）。
- **本地/隐私**：CRG 本地运行（SQLite 图库 + Leiden 社区，零遥测；embeddings / LLM 均 opt-in）；DCP 本地插件；代码不外发。LLM 走公司内网端点（glm-5.2）。
- **生产环境**：Windows + Git Bash（opencode shell 指向 Git Bash）；`~`/`$HOME` 解析为用户目录（`C:\Users\<你>`），`~/.local/bin`、`export PATH`、`~/.bashrc` 等 Linux 写法直接可用，Git Bash 自动给可执行文件补 `.exe`。当前开发在 Linux，命令一致。
- **人在 loop**：设计、提交两处硬检查点。
- **权限最小化**：每个 subagent 只给其职责所需权限。
- **经验沉淀进 wiki**，不灌 `AGENTS.md` / skills。

## 1. 核心思想：双层索引 + 两层上下文策略

### 双层索引：Wiki 导航层 + 代码真相层

把 **wiki** 与 **代码库**当一个**双层索引**协同用——「结合 wiki 与代码库」的主线：

| 层 | 角色 | 特性 | 加载 |
|---|---|---|---|
| Wiki（多级 md） | 导航/语义层：what/why/架构/约定 | 小、稳、便宜 | 先读索引页，按需深读 |
| 代码库 | 真相层：how/实现 | 大、易变、贵 | 经 wiki 的 `source_paths` 锚点 + CRG 代码图，**图引导选择性加载** |

双向锚点：wiki frontmatter `source_paths` → 代码位置；wiki 页之间双链。**三条流都先读 wiki 拿地图，再下钻代码，下钻由 CRG 的图引导只加载相关文件。**

### 两层上下文策略

- **日常层（CRG）**：代码图引导「只加载相关文件」+ CRG 的 `detect-changes --brief` / `visualize --format json` 给紧凑影响面与图，从源头压低上下文体量。
- **兜底层（DCP 全局插件）**：上下文逼近上限时自动压缩（替换为摘要占位，不改会话历史），多数时候 dormant。

> DCP **不是**用来选代码的；选代码由 CRG 图引导。DCP 只在上下文快炸时介入。

## 2. 三条工作流

### WF1 — 代码 → Wiki（分两档）
- **入**：`<repo-path>`、`<wiki-path>`（默认 `<repo>/docs/wiki`）
- **结构页档** `/wiki-map`：`rei` → `cartographer`（`code-review-graph build --repo <R>` 建图）→ `wiki-writer`（`code-review-graph wiki --repo <R>` 生成结构页，同步到 `<wiki-path>` + 补索引）→ 覆盖前 diff 既有 wiki 供审
- **散文架构档** `/wiki-doc`：`rei` → `wiki-writer`（建图+结构页 → 读 Members File 列下钻源码 → 按「职责/组成/工作原理/关键流程(mermaid)/模块关系/注意点/下钻锚点」模板写面向知识库的可读文档）→ 写到 `<out-dir>`
- **出**：多级 markdown wiki（结构页 或 散文架构档，落在 `<wiki-path>` / `<out-dir>`）
- **更新策略**（代码变化后）：
  - 结构页 `/wiki-map`：CRG `update` 增量刷图 + `wiki` 全量重生成（纯结构、快、不调 LLM），覆盖前 diff 供审。无脑重跑即可。
  - 散文架构档 `/wiki-doc`：wiki-writer **按社区增量**——读各页 frontmatter `last_sync_commit` → `git diff <sha>..HEAD --name-only` 拿变化文件 → 用结构页 Members File 列映射 文件→社区 → 只重做受影响社区；未变社区保留旧页。传 `<community>` 则强制单社区覆盖。

> 两档原因见 §5：CRG `wiki` 是**纯结构、不调 LLM**；要拿能进知识库的可读架构文档必须靠 wiki-writer 读源码自己写。

### WF2 — 需求端到端
- **入**：需求文本；repo = cwd 或 `--repo`；wiki 路径
- **流**：`rei` → `planner`（读 wiki + CRG 图定位触点 → 设计：改动点/风险/测试计划）→ 🛑检查点1 → `coder`（按 repo `AGENTS.md` 实现）→ `reviewer`（checklist + `detect-changes --brief` 影响面）→ 循环 → `tester`（gates，按仓发现）→ 循环 → `wiki-writer`（更新受影响 wiki 页）→ `stage` → 🛑检查点2
- **出**：代码改动 + 更新后的 wiki + 变更说明

### WF3 — BUG 定位
- **入**：bug 报告 / 失败代码 / 新代码；repo；wiki
- **流**：`rei` → `bug-tracer`（读 wiki 拿预期行为 + 架构 → CRG 图沿数据/控制流从症状回溯根因 → 根因假设 + 定位代码 + 证据 + 修复建议）→ 🛑报告供审 →（可选）`coder` 施修 → `tester` → `stage`
- **出**：BUG 报告（根因/证据/修复建议）+ 可选修复改动

> 三流都是长任务，可能撑爆上下文——**这才是 DCP 兜底的场景**。

## 3. 成品接线（跨流复用）

| 成品 | 角色 | 形态 | 装法 |
|---|---|---|---|
| **code-review-graph**（CRG） | code_graph_review（图/影响面）+ 结构 wiki（`wiki` 子命令出纯结构页） | CLI `code-review-graph`（`~/.local/bin/`），`--repo <R>` 任意路径 | `uv tool install code-review-graph`（已装 v2.3.7） |
| **opencode-dcp**（DCP） | 上下文兜底压缩 | 全局 opencode 插件，自动 | `opencode plugin @tarquinen/opencode-dcp@latest --global`（已装） |

CRG 子命令（agent 用）：`build` / `update` / `status` / `detect-changes [--brief]` / `wiki` / `visualize --format json` / `serve`（MCP，可选）。本地 SQLite 图库 `.code-review-graph/`，零遥测；embeddings / LLM 均 opt-in。

> 不再自造 deep-wiki / code-graph-review / dcp skill——全由成品覆盖。

## 4. Agent 拓扑 + 权限

primary `rei` 编排 + 人在 loop；其余 `mode: subagent`，最小权限（bash 对 `git *` / `code-review-graph *` 放行，禁 `git commit/push`、`rm`）。

| Agent | mode | 职责 | 关键权限 |
|---|---|---|---|
| `rei` | primary（默认） | 编排三流、检查点、变更说明 | edit: allow, bash: allow（禁 commit/push） |
| `cartographer` | subagent | CRG 建图 + 导出摘要 | edit: deny, bash: 只读 git + CRG |
| `wiki-writer` | subagent | CRG 结构页同步 / **散文架构文档生成**（读源码自己写） | edit: allow（**仅输出目录下**）, bash: 只读 git + CRG |
| `planner` | subagent | 读 wiki+图，出设计 | edit: deny, bash: 只读 git + CRG |
| `coder` | subagent | 实现 | edit: allow, bash: allow（禁 commit/push） |
| `reviewer` | subagent | 自审 + CRG 影响面 | edit: deny, bash: 只读 git + CRG |
| `tester` | subagent | 跑 gates | edit: deny, bash: allow |
| `bug-tracer` | subagent | CRG 图回溯根因、定位 | edit: deny, bash: 只读 git + CRG |

落地：`.opencode/agents/<name>.md`，frontmatter 带 `mode/permission/description`。read-only agents 的 bash 规则：`*`:deny, `git *`:allow, `code-review-graph *`:allow, commit/push deny。

## 5. Wiki 两层产物与 frontmatter

### 两层产物

| 档 | 命令 | 产出方 | 内容 |
|---|---|---|---|
| **结构页** | `/wiki-map` | CRG `wiki` 子命令 | 纯结构：Overview(size/cohesion/lang) + Members 表(前 50 硬截断) + Execution Flows(前 10) + Dependencies(in/out 各 top 15) |
| **散文架构档** | `/wiki-doc` | `wiki-writer` agent | 面向知识库的可读文档：自然语言讲 what/why/how + mermaid 图 + 下钻锚点，不抄源码不纯表格 |

> **CRG `wiki` 真相**：`wiki.py` 纯结构、不调 LLM。`[wiki]` extra 装的是 `ollama>=0.1.0`，但全包 `import ollama` **零命中**——死代码，装了也不加 LLM 摘要。README「LLM summaries」未实现。所以散文档必须靠 wiki-writer（跑在公司 LLM 上）读结构页+下钻源码自己写。截断（Members[:50]、Flows[:10]）是 CRG 设计，无 CLI 参数可解。

### frontmatter 目标字段

```yaml
id / title / level / parent / related(双链) / source_paths(→代码) / last_sync_commit
```

### wiki-writer 每页模板（散文档）

frontmatter + 章节：**职责**（社区定位）→ **组成**（关键类/模块表）→ **工作原理**（按类讲实现+为什么）→ **关键流程**（Execution Flows 文字+mermaid）→ **模块关系**（Dependencies in/out + 边界）→ **注意点**（线程安全/宏/单例/陷阱/合规）→ **下钻锚点**（`文件:行`）。路径一律相对仓库根。

> wiki 根建议结构：`README.md`（入口）+ `architecture/`（L1）+ `modules/<mod>/`（L2/L3）+ `decisions/`（ADR）+ `index.md`。

## 7. 配置（`opencode.json`）

- `$schema`、`default_agent: "rei"`、`instructions: ["AGENTS.md"]`
- `permission`：`edit: ask`；`bash: { *: ask, git *: allow, code-review-graph *: allow, git commit/push: deny, rm *: deny }`；`external_directory: allow`
- `compaction: { auto: true, tail_turns: 15 }`（opencode 自带压缩；DCP 插件在其上做更智能的摘要兜底）
- `model` / `small_model` 由全局配置提供（本仓不写死，指向公司内网 glm-5.2 端点）
- DCP 全局插件已写进 `~/.config/opencode/opencode.jsonc`（`plugin: ["@tarquinen/opencode-dcp@latest"]`）

### DCP 上下文阈值（已按 glm-5.2 调）

DCP 默认 `compress.maxContextLimit=100000` / `minContextLimit=50000` 是**固定数字**，按 ~128K 窗口模型调。glm-5.2 窗口大（GLM-5.x ~200K），固定 50K≈25% 会过早注入软提醒。已在 `~/.config/opencode/dcp.jsonc` 改为**百分比**，跟着真实窗口走：

```jsonc
{
  "$schema": "https://raw.githubusercontent.com/Opencode-DCP/opencode-dynamic-context-pruning/master/dcp.schema.json",
  "compress": { "maxContextLimit": "75%", "minContextLimit": "45%" }
}
```

## 8. 「任意路径」机制

- 工具集在源仓 `/home/dlrow_hl/Rei` 版本化；命令以 `<repo>` / `[wiki]` 路径参接任意目标仓。
- CRG 所有子命令带 `--repo <target>`，不依赖 cwd；agent 用绝对路径操作目标仓，`external_directory: allow` 放行。
- opencode bash PATH 默认不含 `~/.local/bin`；agent 调 CRG 不在 PATH 时用全路径 `$HOME/.local/bin/code-review-graph`（已在 `AGENTS.md` 注明）。
- 重复操作某仓 → 在该仓丢一个 mini `opencode.json`（`references.wiki` + `instructions` 指向该仓 `AGENTS.md`），工具集自动套用。
- （可选）要解锁 CRG 的 MCP 工具，可在目标仓跑 `code-review-graph serve`/`register`；CLI 模式已覆盖三流主路径。
